# Data Model: SaaS Form Submission Platform

**Branch**: `001-form-saas` | **Date**: 2026-03-01

## Overview

Three data entities managed in PocketBase (SQLite):

1. **users** — built-in PocketBase auth collection (no custom fields required)
2. **forms** — user-owned named form endpoints
3. **submissions** — individual POST events recorded for a form

---

## Entity: users

**PocketBase collection type**: Auth (built-in `_pb_users_auth_`)

Using PocketBase's built-in auth collection. No custom fields required for the MVP.

| Field | Type | Notes |
|-------|------|-------|
| `id` | string | Auto-generated by PocketBase (15-char ID) |
| `email` | string | Unique; required; validated as email |
| `password` | string | Bcrypt-hashed by PocketBase |
| `verified` | bool | Email verification status |
| `created` | DateTime | Auto-set on creation |
| `updated` | DateTime | Auto-updated |

**Behaviour**:
- Email verification is enabled via PocketBase's built-in send-verification-email flow.
- Auth tokens are JWT-based (managed by PocketBase).
- Unverified accounts can log in but the Next.js dashboard enforces verification
  before allowing form creation (checked in the auth guard middleware).

## Entity: forms

**PocketBase collection type**: Base

| Field | Type | Required | Constraints | Notes |
|-------|------|----------|-------------|-------|
| `id` | string | auto | PocketBase 15-char ID | Used as the form token in the submission URL |
| `name` | text | yes | min 1, max 100 chars | Human-readable form label |
| `user` | relation (users) | yes | cascade delete | Owning user |
| `telegram_chat_id` | text | no | nullable | Telegram chat ID for notifications |
| `created` | DateTime | auto | — | Creation timestamp |
| `updated` | DateTime | auto | — | Last updated timestamp |

**Access Rules**:
| Operation | Rule | Explanation |
|-----------|------|-------------|
| list | `@request.auth.id = user` | Only the owner sees their forms |
| view | `@request.auth.id = user` | Only the owner views a specific form |
| create | `@request.auth.id != ""` | Any authenticated (verified) user can create |
| update | `@request.auth.id = user` | Only owner can rename or update Telegram config |
| delete | `@request.auth.id = user` | Only owner can delete; cascade deletes submissions |

**Indexes**:
- `user` field: indexed for fast per-user queries (PocketBase auto-indexes relation fields)
- `created` field: indexed for ordering

**Derived data**:
- Submission count is computed at query time via a filter on the `submissions`
  collection (`filter: form = '{formId}'`), not stored as a denormalized field.
  This avoids stale counts and simplifies the data model (Principle V — Simplicity).

## Entity: submissions

**PocketBase collection type**: Base

| Field | Type | Required | Constraints | Notes |
|-------|------|----------|-------------|-------|
| `id` | string | auto | PocketBase 15-char ID | — |
| `form` | relation (forms) | yes | cascade delete | Owning form |
| `data` | json | yes | max total record ≤ 1 MB | All POSTed field k/v pairs; `_next` excluded |
| `created` | DateTime | auto | — | Submission timestamp (used for ordering) |
| `updated` | DateTime | auto | — | — |

**Access Rules**:
| Operation | Rule | Explanation |
|-----------|------|-------------|
| list | `@request.auth.id = form.user` | Only form owner lists submissions |
| view | `@request.auth.id = form.user` | Only form owner views a submission |
| create | `""` (nobody via API) | Created only by PocketBase JS hook (admin-level) |
| update | `""` (nobody) | Submissions are immutable once created |
| delete | `@request.auth.id = form.user` | Owner can delete individual submissions |

**Notes on `data` field**:
- Stored as a JSON object: `{"fieldName": "value", "otherField": "value2", ...}`
- The `_next` field is stripped before storage (FR-009).
- All values are stored as strings (as received from the form-encoded or JSON body).
- No schema enforcement; any field names and values are accepted (open endpoint model).

**Indexes**:
- `form` field: indexed (auto-indexed as relation field)
- `created` field: indexed for reverse-chronological listing

## Entity Relationships

```
users (1) ──────── (many) forms (1) ──────── (many) submissions
         owns                     receives
```

- Deleting a **user** cascades to delete all their **forms**.
- Deleting a **form** cascades to delete all its **submissions**.

## Submission Endpoint Flow (data perspective)

```
External POST to /api/submit/{formId}
    │
    ├─ Find forms record by id = formId → 404 if not found
    │
    ├─ Parse body → extract all fields
    │   ├─ Extract _next field → store value, remove from data map
    │   └─ Remaining fields → { "name": "Alice", "email": "...", ... }
    │
    ├─ Create submissions record:
    │   form = formId
    │   data = JSON.stringify(remainingFields)
    │
    ├─ After successful save:
    │   └─ If forms.telegram_chat_id is set → send Telegram notification
    │
    └─ HTTP response:
        ├─ _next is valid http/https URL → redirect(302, _next)
        └─ otherwise → redirect(302, {FRONTEND_URL}/success?ref={encodedReferer})
```

  "fields": [
    { "name": "name",             "type": "text",     "required": true,  "min": 1, "max": 100 },

## State Transitions

### Form lifecycle

```
[created] → [active, receiving submissions]
          → [telegram_chat_id set] (optional, any time after creation)
          → [deleted] (cascades to all submissions)
```

### Submission lifecycle

```
[received] → [saved] → [notification sent] (if Telegram configured)
                      → [notification failed — logged, submission unaffected]
[saved] → [deleted by owner] (optional)
```
